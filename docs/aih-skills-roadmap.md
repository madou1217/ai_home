# aih Inner Skills 路线图（按必要性优先级）

> 2026-07-04。前提认知：aih 的独有资产 = **多 provider 网关 + 账号池 + 能力目录 + 用量体系 + 持久会话**。
> inner skill 的判定标准：**只有站在 aih 网关上才做得到、且是会话内 agent 的真实痛点**。
> 不满足这条的（如 web 搜索、通用工具）一律不做——各 CLI 自带或与 aih 无关。
>
> 已交付：`aih-collab`（跨 provider 识图/出图借用，2026-07-04，commit b7100ee）。
> 本文档所有 skill 复用同一套基建：`assets/provider-skills/` 资产 + `provider-skill-installer` 软链/受管块安装 + `AIH_GATEWAY_BASE_URL/API_KEY` 会话 env + `/v1/models?capability=` 能力发现。

---

## P0 — 必要（直接堵正在流血的痛点）

### 1. `aih-usage-aware` — 用量与配额自觉
**痛点（刚发生的实证）**：2026-07-04 opencode 唯一账号撞 5 小时限额，会话里的 agent 对此**一无所知**——只会把 429 原文甩给用户，不知道是哪个账号、多久恢复、有没有别的路。用户的第一反应是"是不是切换 bug 又犯了"，排查成本全落在人身上。
**skill 内容**：
- 长任务/大批量调用**开工前**：`GET /v0/webui/accounts`（或专设轻量端点）查当前 provider 账号余量与冷却状态，余量告急先告知用户再动工。
- 收到 429/配额类错误时：查账号状态 → 向用户翻译成人话（"opencode 唯一账号 5 小时限额，6 分钟后恢复；claude-* 别名会自动落到 agy 兜底"），并给出选项（等待/换模型/换 provider）。
**依赖基建**：usage snapshot、账号状态 API、runtime cooldown 字段——**全部已存在**，零网关改动。
**成本**：低（纯 SKILL.md + 可能一个脱敏只读端点）。
**为什么 P0**：撞限额是多账号网关的日常，agent 的"错误翻译官"角色能消灭一整类"是不是 bug"的排查；与刚修的 probe-recovery/429 冷却机制形成闭环。

### 2. `aih-second-opinion` — 跨模型第二意见 / 会诊
**痛点**：agent 卡壳（调试兜圈、方案拿不准、自我评审盲区）时只能一条道走到黑。aih 网关背后坐着 claude/gpt/gemini/glm 全家——这是单 CLI 永远没有的资产。
**skill 内容**：
- 触发条件写严：同一错误连续 N 次修不好、重大不可逆决策前、用户明确要求 review 时。
- 打包"问题 + 关键上下文（≤N KB）"POST 给**不同家族**的模型（能力发现选 `owned_by` 不同者），把回答作为参考意见综合，明确标注"X 模型的意见"。
- 反滥用规则：每任务限 1-2 次，禁止把它当主推理引擎。
**依赖基建**：/v1 chat + aih_modalities——**全部已存在**，零网关改动。
**成本**：最低（纯 SKILL.md，文案工程为主）。
**为什么 P0**：实现最薄、普适性最强（每个 provider 每种任务都受益），是"网关价值"最直接的兑现。

## P1 — 高价值（明确场景，少量基建扩展）

### 3. `aih-longcontext` — 长上下文托管阅读
**痛点**：glm-5.2 等小上下文模型面对大文件/长日志直接歇菜；而池子里有 gemini 1M 级上下文模型闲着。
**skill 内容**：文件超过自身处理能力时，把大文件发给长上下文模型做"定位式阅读"（找出与问题相关的段落/行号/摘要），拿回浓缩结果继续干活。强调**定位而非转述**（返回行号与原文引用，防二手失真）。
**依赖基建**：需要把 models.dev 的 `limit.context` 纳入 modality 索引并暴露（`/v1/models` 加 `aih_context_length`，`?capability=long_context`）——**modality-index 顺手扩展，~半天**。
**为什么 P1**：场景清晰频繁，但比 P0 少一分"不做就流血"的紧迫性。

### 4. `aih-collab` v2 — PDF / 图片编辑扩展
**痛点**：现版只覆盖"识图 + 文生图"。models.dev 元数据里 claude 系 input 含 `pdf`；gemini image 系支持图生图（带参考图编辑）。
**skill 内容**：PDF 理解借用（发 data URL 给支持 pdf 的模型）；图片编辑（用户给图+改图要求 → image 模型带 image_url 输入）。
**依赖基建**：`?capability=pdf` 过滤（索引已有数据，加一个 capability 枚举）；图生图链路需真机验证 code-assist 路径对"输入图+输出图"的支持。
**为什么 P1**：现有 skill 的自然延伸，边际成本低；但 pdf/改图的需求频率低于识图/出图。

## P2 — 进阶（价值成立，等 P0/P1 用出反馈再做）

### 5. `aih-consensus` — 关键决策多模型投票
**内容**：高风险决策（数据迁移方案、安全设计）并行问 3 个不同家族模型，汇总分歧点呈给用户。与 second-opinion 的区别：并行、多数决、只用于用户明确授权的重决策。
**克制理由**：token 成本 ×3，滥用风险高；先让 second-opinion 验证"跨模型咨询"的实际效果再升级。

### 6. `aih-cost-router` — 成本自觉的子任务降级
**内容**：agent 把机械性子任务（大批量分类/格式判断/简单抽取）主动委托给池中便宜快速的模型，把贵模型配额留给核心推理。
**依赖**：模型成本元数据暴露（models.dev 有价格数据，索引未暴露）；以及"agent 判断子任务可降级"的文案工程——这是难点，写不好会答非所问。
**克制理由**：与网关 alias/路由的职责有重叠区，先观察 P0 用量自觉是否已缓解配额压力。

## P3 — 远期（依赖深度集成，暂缓）

### 7. `aih-session-handoff` — 跨 provider 会话交接
**内容**：当前模型配额尽/能力不足时，agent 生成结构化交接摘要（目标/已完成/未完成/关键决定），通过 aih 会话体系在另一 provider 开新会话续跑。
**暂缓理由**：交接质量强依赖摘要文案；且与 WebUI 会话/registry/native-session 深度耦合，值得做但需要独立设计（近似"Fabric 北极星"里的跨节点协作命题，应与之合并考虑）。

### 8. `aih-fleet-status` — 集群/节点感知（Fabric 侧）
**内容**：agent 可查 Fabric 远程节点状态、在远端节点上发起任务。
**暂缓理由**：Fabric 主线（Node+能力标签）尚在收敛，skill 应等主线 API 稳定后跟进，避免双向返工。

## 明确不做（记录否决理由，防止重复讨论）

| 提案 | 否决理由 |
|---|---|
| web 搜索/浏览器代理 skill | 非 aih 独有能力，各 CLI 自带方案；网关不是搜索引擎 |
| embedding/语义检索 skill | aih 无向量基建，新增整套 RAG 违背"网关"定位 |
| 通用"工具箱"型 skill（格式转换等） | 与 aih 资产无关，agent 自己写脚本即可 |

## 横切原则（适用于所有新 skill）

1. **触发条件写严**：每个 skill 必须有"何时不要用"段落——skill 是兜底通道，不是主路。
2. **诚实原则**：网关不可达/无可用模型/借用失败必须明说，禁止虚构结果（aih-collab 已立此规矩）。
3. **一次一借**：禁止对网关循环重试（429 事故的教训——重试风暴打限额上游）。
4. **能力发现优先**：一律 `?capability=` 动态发现，禁止硬编码模型名（模型池是流动的）。
5. **每个 skill 独立目录**，共用 installer；安装面变化（新 provider）改 installer 不改 skill。
